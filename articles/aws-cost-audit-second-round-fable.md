## はじめに

[前回の記事](https://zenn.dev/koedesk/articles/aws-cost-reduction-with-claude-code)で、Claude CodeでAWSの月額費用を約$5,000削減した話を書きました。今回はその続編です。

きっかけは正直、しょうもない話です。Claudeの新モデル「Fable 5」が使えるようになって、サブスクリプションの週次トークン制限が余っていた。せっかくの上位モデルの枠を余らせるのはもったいない。**何か有用なことに焼き切れないかな**、と考えて思いついたのが「AWS費用のゼロベース監査」でした。

前回あれだけ削減したので、正直もう大きな玉は残っていないと思っていました。結果、半日のセッションで**月$8,000規模の削減案**が出てきました。びっくりでした。

| 発見 | 削減見込み/月 |
|------|-------:|
| Savings Plans 3年契約（前払いゼロ） | $1,800 |
| 監視・テレメトリの二重装備と全量送信 | $1,700 |
| 「消したつもりで消えていなかった」S3ログ 62TB | $1,400 |
| 封じられていたDynamoDBオートスケーリング | $1,300 |
| NATと二重装備のVPCエンドポイント群 | $1,100 |
| 外し忘れの監査ログ（2年間稼働） | $400 |
| 1年間で利用実績1分の調査ツール | $250 |
| **合計** | **約$8,000** |

※ 一部はインフラチーム確認中の提案ベースなので、満額で着地しない可能性はあります。それでも当社のAWS費用は月$24,000ほどなので、通れば3分の1が消える計算です。

前回が「夜間停止」「VPN移行」のような**足し算で見つかる削減**だったのに対して、今回見つかったのはほぼ全てが「**設定はあるのに、効いていない**」「**動いているのに、誰も使っていない**」という類のものでした。人間の目grep では、まず見つからないやつです。

## 「効いているつもり」シリーズ3連発

今回の監査で一番ぞっとしたのが、この3つです。どれも管理画面上は「対策済み」に見えます。

### 1. ライフサイクルが設定されているのに、1バイトも消えていないS3

あるログバケット群のストレージが毎月10TBペースで増え続けていました。おかしいのは、**「3日で削除」のライフサイクルルールがちゃんとEnabledで存在していた**ことです。

原因は3つの設定の組み合わせでした。

1. バケットのバージョニングが有効 → Expirationは実データを消さず「削除マーカー」を置くだけ
2. `NoncurrentVersionExpiration` に `NewerNoncurrentVersions: 1`（最新の非最新バージョン1個は保持）が設定されている
3. ログは一度書いたら上書きされない → **各オブジェクトの非最新バージョンはちょうど1個**

つまり全オブジェクトが「保持対象の1個」に該当し、**永遠に誰も削除されない**。半年前のログが削除マーカーの裏に62TB積もっていました。コンソールの通常表示では削除マーカー越しに見えないので、気づきようがない。

Fable 5はこれを、費用の増加トレンド → バケット別サイズの時系列 → ライフサイクル設定の読解 → 「古いオブジェクトが非最新バージョンとして現存する」実証、という順で特定してきました。ライフサイクルの仕様の噛み合わせまで読んで「構造的に削除ゼロになる」と言い当てたのは、正直感心しました。

### 2. Fargate Spot化、実は効いていなかった

前回の記事で「Fargate Spot化で$263/月削減」と書きました。すみません、**あれは幻でした**。

今回の監査で請求をusage type単位で見たところ、Spotの使用実績が通年でほぼ$0。クラスタ設定を確認すると、キャパシティプロバイダにFARGATE_SPOTが**アタッチすらされていない**。設定した形跡が一瞬だけ請求に残っていたので、おそらくIaC（Terraform/CloudFormation）の再適用でコンソール変更が巻き戻されたのだと思います。

教訓は2つ。**手で入れた設定はIaCに勝てない**こと。そして**「やった」と「効いている」は請求で確認するまで別物**だということです。

### 3. オートスケーリングの最小値が、確保量と同じ値になっていた

DynamoDBのプロビジョンドテーブルで、Target Trackingオートスケーリングは設定済み。でも費用は一定。設定を開いたら、

> Min: 1000 / Max: 1100（現在の確保量: 1000）

**下限が現在値に張り付いている**ので、自動調整は生きているのに下げる方向には1ミリも動けない。エアコンの自動運転で最低温度を30℃に設定しているようなものです。実消費を7日分測ったら平均利用率は20〜30%、市場が休みの週末はほぼゼロ消費で満額課金でした。

Minを実態に合わせて下げるだけ（Maxは据え置きなので急騰時の天井は変わらない）で、4桁ドルが浮きます。

## 動いているのに、誰も使っていないもの

### 1年間で利用実績「1分」の調査ツール

Amazon Detective（GuardDutyの事後調査ツール）が1年前から有効で、月$250かかっていました。

面白いのはここからで、「そもそも誰か使ったことあるの？」という問いに、**CloudTrailでDetective APIの呼び出し履歴を照会する**ことで答えが出ます。直近90日で人間のアクセスは、たった1回・約60秒のセッションのみ。しかも同時に「無料トライアル状況の確認」APIが呼ばれていたので、調査ではなく「これ何だろう」と覗いただけと推定できる。調査機能の実行履歴はゼロ。

**年$3,000払って、利用実績は覗き見1分**。こういう「使っているか分からないもの」の裏取りに、CloudTrailの操作履歴照会はとても効きます。おすすめです。

### 2年間動き続けていたデバッグ用の監査ログ

キューサービスの**全データイベントを記録する監査設定**が2年前から3環境で動いていて、月$400。中身を見ると毎秒100件以上、ほぼ全てが同じアプリロールによる定型的なキュー操作の記録でした。

開始時期の並び（開発環境が先、本番が2ヶ月後）から見て、当時の障害調査で入れたデバッグ設定の外し忘れがほぼ確実。**監査ログは「入れた日」より「消す日」を決める方が難しい**、という典型例です。

### NATと二重装備のVPCエンドポイント

NAT Gatewayがあるのに、Interface型VPCエンドポイントを10個以上フル装備しているVPCが複数ありました。エンドポイントは1個あたり約$10/月×AZ数なので、10個×3AZで$300/月。それが複数アカウントに同じ構成で入っていて、合計4桁ドル。

消して大丈夫な根拠も面白くて、**ECRのコンテナイメージ本体はS3に置かれているため、無料のS3 Gatewayエンドポイント経由で流れる**。つまりInterface型を消してもNATに乗る通信はわずかで、データ処理費の増加は誤差でした。この「どの通信がどの経路に乗るか」の突き合わせをモデルが淡々とやってくれるのは、かなり体験が良いです。

## 一番実用的だった学び: Savings Plansは「期間」が全てで「払い方」はおまけ

前回、コンピュートのSavings Plans購入を「1年分の前払いで$36,000」という計画で温めていました。今回Fable 5がCost ExplorerのレコメンドAPIを叩いて4パターンを並べたら、こうなりました。

| | 前払い | 毎月の削減 |
|---|-------:|-------:|
| 1年 No Upfront | ¥0 | -$707（14%） |
| 1年 All Upfront | 約$43k | -$1,020（20%） |
| **3年 No Upfront** | **¥0** | **-$1,841（36%）** |
| 3年 All Upfront | 約$88k | -$2,160（43%） |

見てほしいのは2点です。

1. **期間を1年→3年にすると割引が2.5倍に跳ねる**のに、**払い方を月払い→一括前払いにしても数ポイントしか変わらない**。3年契約同士で比べると、$88kの現金を3年寝かせて得られる上乗せは月$319——年利換算で4%程度でした
2. **「3年の月払い」は「1年の全額前払い」に、前払いゼロで勝つ**

つまり「まとまったキャッシュができたら前払いで買おう」とタイミングを待っていたこと自体が間違いで、**待っていた1ヶ月ごとに$1,841を余分に払っていた**ことになります。前払いゼロなら意思決定した瞬間に効き始める。この整理が出た日のうちに社内Slackで合意が取れて、購入に進みました。

（もちろん3年コミット自体のリスクはあります。うちの場合は対象ワークロードの大半が特定事業に乗っているので、「その事業が何ヶ月続けば損益分岐か」まで計算させました。答えは16ヶ月でした。この手の試算が会話の速度で出てくるのが、監査をAIにやらせる一番の効能かもしれません）

## 進め方: AIは「提案と証拠」まで、判断は人間とチーム

誤解のないように書いておくと、Fable 5が勝手にリソースを消して回ったわけではありません。今回のセッションでモデルがやったのは、

1. Cost Explorerで費用をサービス別・アカウント別・usage type別に分解
2. 増えているもの・不自然なものについて、実リソースの設定を読みに行く
3. 「本当に使われていないか」をCloudTrailやメトリクスで裏取り
4. 削減案を、**根拠の実測値つきでインフラチームへの確認チケットに起票**（「この設定を入れた意図はありますか？要件があればこのままクローズで構いません」という聞き方で）

というところまでです。消す・変える判断は、経緯を知っているチームがやる。監査ログのように「実は要件があった」となれば、それはそれで文書化されるので無駄になりません。

## お金の勘定

今回のセッションは、追加のAPI費用ゼロ——**もともと契約しているサブスクリプションの、余っていた週次枠だけ**で完結しています。使ったのは半日ぶんの対話と、AWSの読み取り権限だけです。

前回の記事で「Claude Codeの導入コストは増えたが、それ以上をClaude Code自身が回収してくれた」と書きました。今回はそれがさらに進んで、**シート1人分（月$150）のツールが、暇つぶしに月$8,000の削減案を持ってきた**格好です。AIツールの費用対効果の議論はいろいろありますが、少なくとも「余った枠でインフラ費用を監査させる」は、確実に元が取れる使い方だと思います。

## おわりに

2回のコスト削減をやってみて思うのは、クラウドのコストの敵は「高いサービス」ではなく、**「効いているつもりの設定」と「誰も覚えていないリソース」**だということです。どちらも管理画面を眺めているだけでは見つからず、請求データ・設定・操作履歴の3つを突き合わせて初めて見えてきます。そしてその突き合わせ作業は、人間よりAIの方が圧倒的に得意で、圧倒的に飽きません。

週次トークン、余っていたら焼き切りましょう。おすすめはAWSの請求です。
