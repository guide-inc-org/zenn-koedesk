組織の全リポジトリで AI による Pull Request レビューを走らせる、というのを 1 週間で立ち上げました。

前提として、当社の Organization には **リポジトリが約 200** あります。BFF、ネイティブアプリ、仕様書リポジトリ、インフラ、ライブラリ、各種ツール…まあ普通の Web 企業のスケールではあると思います。

この規模で AI による PR レビューを入れようと考えたとき、最初から避けたかったことが 1 つありました。

**`.github/workflows/claude-pr-review.yml` を 200 リポジトリに配って回ること**です。

仮にうまく配れたとしても、観点を更新するたびに 200 リポジトリに同じ修正を配って回ることになりますし、新規リポジトリができるたびに「あ、これにも入れといて」のリマインドが要ります。考えるだけで嫌になります。

なので最初から、**中央の 1 リポジトリにポンと置いたら、それが他の全リポジトリにも効くような仕組み**を取りたい、というのがスタート地点でした。

最終的にできた構成は、**各リポジトリには workflow ファイルを1つも置かない**、というものです。スイッチは GitHub のカスタムプロパティ 1 個だけ。`claude_pr_review=enabled` を立てたリポジトリには中央から自動的にレビューが配信されて、外したら止まる。

この記事は、その構成と、立ち上げ中に踏んだ「カスタムプロパティの伝播遅延」「中央 workflow から org secret は読めるのか？」「多言語チームでレビューを何語で書かせるか？」あたりの実運用の話をまとめたものです。

## やりたかったこと

要件は 2 つだけです。

1. **各リポジトリに workflow ファイルを置かずに、全リポジトリで AI レビューを動かしたい**
2. **新しいリポジトリができたら、スイッチひとつで有効化したい**

最初は「200 リポジトリで yml を配り歩く未来から逃げたい」くらいの動機でした。後述しますが、結果的にこれが運用面でいろんな副次効果を生むことになります。

## 構成: Custom Properties × Ruleset × Bedrock 経由 Claude

最終的にたどり着いた構成です。

```
[ 中央リポジトリ ]
  .github/workflows/claude-pr-review.yml   ← 唯一の workflow 本体
            │
            │ (Repository Ruleset で配信)
            ▼
[ Custom Property: claude_pr_review = enabled が立っている全リポジトリ ]
            │
            │ pull_request イベントで発火
            ▼
[ AWS Bedrock 上の Claude にレビューさせる ]
            │
            ▼
[ PR にレビューコメントが付く ]
```

3 つの仕組みを組み合わせています。

1. **GitHub Organization Custom Properties** — 各リポジトリに「このリポジトリは AI レビュー対象か？」を表すメタデータを付けます
2. **Repository Ruleset** — `claude_pr_review=enabled` のプロパティを持つリポジトリ群を対象に、中央リポジトリの workflow を強制実行する
3. **AWS Bedrock 経由の Claude** — Anthropic 直契約ではなく自社の AWS アカウントから呼ぶことで、IAM 管理 / ログ集約 / コスト集約をすべて自社側に閉じる

この組み合わせの効果は、各リポジトリには `.github/` 配下に **1 ファイルも置かなくていい**ということです。

新規リポジトリを作る → カスタムプロパティに `claude_pr_review=enabled` を入れる → それで終わり。レビューが翌 PR から走り始めます。

## 実装の中身

### 1. Custom Property を Org レベルで定義する

Organization の Settings → Custom properties で、`claude_pr_review` というプロパティを `single select` で定義し、値を `enabled` / `disabled` の 2 つにしました。

### 2. Repository Ruleset で「対象リポジトリ」を絞る

Organization の Rulesets で、新しいルールセットを作り、conditions に以下を設定します。

```yaml
target: repository
conditions:
  ref_name: ~ALL
  repository_property:
    include:
      - property_name: claude_pr_review
        property_values: [enabled]
```

これで「`claude_pr_review=enabled` を持つ全リポジトリ全ブランチ」が対象になります。

### 3. 中央リポジトリに workflow 本体を1つだけ置く

中央リポジトリ (例: `org/.github`) の `.github/workflows/claude-pr-review.yml` に、PR トリガーで Bedrock 経由で Claude を呼ぶ workflow を1本だけ書きます。

ここがポイントですが、**Ruleset で配信される workflow は、対象リポジトリの実行コンテキスト下で走ります**。なので、対象リポジトリの PR ペイロードがそのまま使えますし、`secrets.GITHUB_TOKEN` も対象リポジトリ用のものが自動的に発行されます。

### 4. AWS Bedrock 側の準備とモデル選定

GitHub Actions から AWS への認証は OIDC を使った AssumeRole で行います。Bedrock の対象モデルを有効化して、IAM ポリシーで `bedrock:InvokeModel` を付けるだけです。API キーを GitHub Secrets に置く必要がないので、ローテーション運用が要りません。

モデルは **Claude Sonnet (4.6) を日本リージョンで** 使っています。

正直に言うと、最初は Opus を使いたかったんです。レビューの質も体感で違いますし。ただ、前述のとおりリポジトリが約 200 ある世界で、それぞれが毎日数本ずつ PR を出すと、Opus で全部レビューさせるのはコスト的にちょっと怖い。

なので甘んじて Sonnet で運用しています。Sonnet でも十分に役に立つ指摘は出てくるので、現状はこのトレードオフで満足です。

## 個人的に一番面白い使い方: 「過去の本番障害」をレビュー観点に変換する

ここからが、この仕組みで一番気に入っている部分です。

AI レビューはただ「コードをきれいに見てくれ」と頼んでも、当たり障りのない一般的な指摘しか出ません。LGTM もどきが量産されるだけです。

そこで中央 workflow には、レビュー観点のチェックリストをプロンプトに同梱しています。最初は「セキュリティ観点」「パフォーマンス観点」みたいな教科書的なものから始めましたが、運用していくうちにここに **「過去に起きた本番障害」を仕込む** ようになりました。

具体的にはこういう運用です。

- 過去に本番障害を起こした修正パターン (例: 環境変数の追加漏れ、特定の認証フラグの取り扱いミス、規模を考慮しないクエリの混入など) を、観点として 1 行ずつチェックリストに追記する
- 本番障害の起因となったファイルや関数を「触ったら警告」のターゲットとして登録する
- PR にそれらが含まれた瞬間、Claude のレビューが「ここ、過去に踏んだ地雷の近くです」と教えてくれる

これがやりたかったことです。

「再発防止策」って、世の中で書かれているもののほとんどが **「チェックリストを作ります」「気をつけます」「ダブルチェック体制を強化します」** みたいな、ふわっとした宣言で終わりがちです。書いた当人もたぶん後で読み返さないし、別のメンバーが踏んだら何の意味もない。正直、形だけの再発防止策は多い。

その「形だけの再発防止」を、**毎回の PR で機械的に走るチェック** に変換できます。

ポイントは、**障害を起こした本人だけじゃなく、別のメンバーがそのコードを触ったときにも、ちゃんと警告が出ること**です。組織のナレッジが個人の記憶じゃなくて、中央リポジトリの 1 ファイルに集約されて、毎回の PR レビューに自動的に注入される。

実効性が本当にあるかどうかは、正直まだ検証中です。AI が拾いきれない地雷はもちろんあります。ただ「再発防止策を書類で書いて終わり」よりは、PR の上で 1 行コメントとして毎回目に触れるほうが、はるかにマシだと個人的には思っています。

少なくとも、面白い試みではあります。

## ハマりどころ 3 つ

立ち上げ中に踏んだ罠を 3 つだけ書いておきます。同じ構成をやる人の役に立つはずです。

### ① Custom Property の伝播に最大 30 分かかる

`claude_pr_review=enabled` を設定しても、Ruleset 評価器が「あ、このリポジトリも対象だ」と認識するまでに時間差があります。経験則で **5〜30 分**。

なので、PoC で「設定したのに動かない！」と慌てる前に、まずコーヒーを淹れに行きましょう。

これは事前にプロパティだけ立てておいて、後から workflow 本体や Ruleset を整える運用のほうが効率的、ということでもあります。

### ② 中央 workflow から org secret は「読めます」

これは事前に確信が持てず実機検証しました。結論、**Ruleset 経由で配信された workflow からも、対象リポジトリが許可されている org secret は読めます**。

なので、Bedrock 用の AWS OIDC ロール ARN や、Slack 通知用の Webhook URL を org secret に置いておけば、全リポジトリで使い回せます。

### ③ レビュー言語をどうするか

最初はプロンプトを「PR の説明文に合わせた言語でレビューしてくれ」としていました。これだと PR タイトルが英語でも本文が日本語のリポジトリでは挙動が不安定になります。

最終的に当社では、**日本語とベトナム語を必ず両方併記する** ことに倒しました。

というのも、当社は日本に本社を持ちつつ **ベトナムにオフショア拠点** を構えていて、私自身もそのベトナム拠点側にいる人間です。日本側のメンバーとベトナム側のメンバーが、同じ PR をそれぞれ自分の言語で読めるようにしておくのが、結果的に一番揉めごとが少なかったです。

2 か国だから「日本語とベトナム語」でシンプルに済みました。ただ、これが 3 か国 4 か国と増えた組織だと、ここの設計はもっと頭を使う気がします。

**多言語でオフショア展開している組織の方が AI PR レビューをどう運用しているのか、ぜひ教えてほしいです。** 私もいま、ここは正解を探しているところです。

## 効果

立ち上げ初週で、まずパイロット的に **12 リポジトリ** に展開しました。

その後、社内に「こういう仕組みを入れたよ」と展開したところ、**「うちのリポジトリにもオンにしてくれ」という依頼が続々と入ってきて**、執筆時点では約 200 リポジトリのうち **27 リポジトリ** で稼働しています。短期間で 2 倍以上になりました。

これは個人的にも嬉しい誤算で、「強制配信」というと一見トップダウンっぽいですが、実際にはチーム側から「うちにも入れて」と手挙げで来てくれるんだなと。

新規リポジトリができたとき、もしくは既存リポジトリで「AI レビューつけてほしい」と言われたとき、私がやる作業は次の 1 コマンドだけです。

```bash
# 専用スキルでカスタムプロパティを立てる
claude /claude-pr-review-enable <repo-name>
```

リポジトリには 1 ファイルも触りません。PR も作りません。承認も要りません。プロパティを立てるだけです。

「PR レビュー欲しい」と言われてから、実際に動き始めるまで **30 秒** です。

## まとめ

`.github/workflows/` を各リポジトリに配って回る運用は、リポジトリ数が増えれば増えるほど指数関数的にダルくなります。Reusable Workflow はそのつらさを和らげますが、根本的には「呼ぶ側に何かを置く必要がある」という制約から逃げられません。

Custom Properties × Repository Ruleset × Bedrock 経由 Claude の組み合わせは、この制約自体を取り払います。

**配るのをやめて、メタデータで配信対象を絞る**。

これは workflow の話ですが、根っこの発想は「リポジトリの中の設定を中央に持ち上げる」というガバナンス設計の話でもあります。たぶん同じパターンが、他にもいろいろ応用できます (Backlog 連携の自動投稿、ミラーリング先の自動切り替え、リリースノート生成の対象選定など、いくつか手元で実験中です)。

そして、最初は「ただ管理を楽にしたかった」だけで始めたこの構成が、**過去の本番障害を毎回の PR で機械的にチェックする** という、書類で書いて終わりがちな「再発防止策」の実効化につながったのは、自分でも予想していなかった副産物でした。

組織で AI PR レビューを入れたい、けど各リポジトリに workflow を配って回るのはしんどい、という人にこの構成は刺さると思います。

スイッチひとつで全リポジトリを AI レビュー対象にできる世界、悪くないですよ。
