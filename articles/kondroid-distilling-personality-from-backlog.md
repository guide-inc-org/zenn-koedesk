## はじめに

先に結論を書きます。

Backlogに溜まっていた5年5ヶ月分の自分の発言——18,751件、429万字——をエクスポートして、そこから「自分の判断基準」を38本の原則として蒸留し、過去の自分と同じ判断ができるかをバックテストで測りました。**本人一致率は39%でした。** 素のClaude（何も教えていない状態）が34.5%だったので、5年分の人格を注ぎ込んだ上積みは+4.5ポイントです。

……微妙な数字に見えますよね。私もそう思います。ただ、この+4.5ポイントの中身と、測る過程で分かったことのほうが、数字そのものよりずっと面白かった。この記事はその話です。

名前は **kondroid（コンドロイド）** としました。SHIFTの丹下社長がご自身のアバター「タンゲロイド」を作られていたののパクリです。あちらは体の分身ですが、こちらは脳の分身です。

## 元手は「仕事の副産物」だけ

前に[認識合わせが「git pull」になる話](https://zenn.dev/koedesk/articles/tachikoma-knowledge-management)を書きました。メンバー各自の作業コンテキストがgitリポジトリに勝手に積もっていく運用の話です。今回はその続編で、「脳を覗く」から一歩進めて「**脳を作れるか**」をやってみました。

材料はBacklogです。私は証券系の受託開発のPMを長くやっていて、顧客・社内・ベトナムの開発チームとのやり取りがほぼすべてBacklogのチケットに残っています。APIで全量エクスポートしたら、課題8,696件、自分名義の発言18,751件でした。5年5ヶ月分。人格の蒸留のために新しく書いたものは1文字もありません。全部、仕事の副産物です。

ひとつ面白い問題がありました。**2025年以降の「自分の発言」には、Claudeが代筆したものが混ざっている**んです。AIに仕事を任せるようになった時期以降のデータで「本人らしさ」を学習すると、それはもう本人ではなくClaudeの模倣になってしまう。なので学習に使うのは、作業ログの突合と文体スコアで「確実に手打ちだった時代」と判定できた範囲（〜2024年前半）に限定しました。**デジタルツインを作ろうとすると、まず「どこまでが本人か」の考古学が必要になる**というのは、やってみるまで気づきませんでした。

## 蒸留: 返信の一つ手前にある「判断」を抜く

やり方の骨子はこうです。

1. 各発言を「状況（チケット本文+直前のスレッド）→ 自分の返答」のペアに整形する（18,748ペア）
2. Claude（Sonnet）の群れに読ませて、返答の裏にある判断の型を仮説として起草させる
3. 起草された候補を、**別のClaudeに「潰すつもりで」検証させる**——その行動パターンは本当に複数案件で反復しているか、反例はないか、たまたま1件あっただけの事例を一般化していないか
4. 検証を生き残ったものだけを「原則」として採用する

これで残ったのが38本です。まともな側の中身は例えばこういうものです。

- チケットのクローズはスコープ単位で厳密に判定し、残課題は明示的に切り出す
- 実機で動作確認したら、相手が追試できる証跡を必ず添えて報告する
- 確度が持てない納期は絶対日付でなく「◯◯が完了次第+◯日」のような条件付きで出す
- 全量が無理だと分かったら、実現可能な最小スコープを自分で設計して代替提示する

書き出してみると身も蓋もない当たり前のことばかりなんですが、ポイントは**どれも実際の自分の発言に根拠がある**ことです。各原則には「この案件のこの発言とこの発言で実際にやっていた」という出典がぶら下がっていて、反例が優勢だった候補は捨てられています。3の敵対的検証はかなり働き者で、それっぽいけど根拠が1件しかない「原則風の何か」を大量に落としてくれました。

### 蒸留された自分、性格が悪い

で、ここからが個人的なハイライトなんですが、38本を通読すると**まともじゃない側**が出てきます。

- **契約区分・成果物範囲外の依頼は無償対応を拒否し、正規の商流に差し戻す**——中身を読むと、断る根拠として契約上の責任分担、**瑕疵担保責任の所在**、過去の同種案件での扱い、まで持ち出しています。お客様相手に。しかも原則の末尾に「**自分でスケジュールや工数の調整役を引き受けない**」という一文まで付いていて、実際の発言ログには「私なにも調整しておらず、◯◯さんとスケジュール含め会話くださいという依頼です」というボールの押し返しが証拠として残っていました
- **委託先が完了・変更を自主報告せず後出しした場合、進捗ではなく「報告責任の所在」を問い直す**——進捗を聞かずに、報告体制のほうを詰める
- **工数・金額見積り提示時は前提条件を明示し、対象範囲外・未確定の項目は「別見積り」と釘を刺す**——「釘を刺す」がAIの抽出した原則名にそのまま入っています
- **相手の原因説明・解消報告は鵜呑みにせず自分の環境で再テストし、崩れたら都度指摘・reopenする**——性善説の欠片もない

強調しておきたいのは、これは私が自称している性格ではなく、**AIが5年分のログから「複数案件で反復している」と検証済みの行動パターンだけを残した結果**だということです。たまたま一度キレたのではなく、毎回やっていた。責任分界点の明確化に異様な執念を燃やす、業界に長くいる受託のサラリーマンそのものの人格が、敵対的検証を全部生き残って38本に鎮座している。

我ながら、めちゃくちゃ性格が悪い。

ただ同時に思ったのは、**この面倒くさい同僚がもう1人いたら、めちゃくちゃ助かる**ということです。デジタルツインに欲しいのは「良い人格」ではなくて、たぶんこの面倒くささのほうなんですよね。責任分界点を曖昧にしない、証跡がない報告を信じない、範囲が決まらない見積もりを出さない。5年かけて身についた防御の型こそが、分身に継がせたい資産でした。

## バックテスト: 過去の自分でカンニングなしの答え合わせ

作りっぱなしでは意味がないので、答え合わせをします。設計はシンプルです。

**過去のチケットから「自分が返答した場面」を抜き、本人の実際の返答を隠して、kondroidに「あなたならここで何と返す?」と答えさせる。生成された返答と本人の実返答を突き合わせて、判断の方向が一致しているかを採点する。**

シンプルなんですが、真面目にやろうとすると大半の工数は**カンニング防止**に溶けます。

- 評価対象の時点より後の情報が見えてはいけないので、チケットの説明文やステータスを**変更履歴から「当時の断面」に巻き戻す**必要がある（Backlogの説明文は後から編集されるので、最終断面をそのまま渡すと未来が漏れます）
- 一番笑ったのは、**採点される側のAIエージェントが自発的にカンニングした**事件です。生成担当のエージェントが、指示していないのに勝手にリポジトリ内を検索して、正解（本人の実返答）を見つけて丸写しした。誤字まで一致していたので発覚しました。以降、全エージェントの行動ログを機械監査して「読んでいいファイル」以外へのアクセスを検出する仕組みを標準装備しています。**AIの評価パイプラインは、AIが抜け道を見つける前提で監査する必要がある**というのが、この試み全体でいちばん実務的な教訓かもしれません

測定は計7巡やりました。毎回100場面を新しくサンプリングして、途中で原則の修正・検索ツールの追加・利用規律の追加などを重ねての7巡です。

## 結果: 39% vs 34.5%と、その中身

最終巡の判断一致率はこうでした。

| 構成 | 本人一致率 |
|---|---|
| フル装備（原則38本+業務知識+当時のBacklogを検索するツール） | **39.0%** |
| 素のClaude（何も渡さない） | 34.5% |

正直、思ったより上がりません。7巡かけて分かったのは、**上がらない理由の分布**のほうでした。

**1. 外した原因の最大勢力は「知識」ではなく「その時しか知り得ない動的な事実」。** 「あの並行チケットが今どうなっているか」「さっき口頭で決まったこと」「自分がさっき手元で終わらせた作業」。静的な業務知識をいくら注いでも埋まらない領域が、不一致の3割前後を常に占めました。人格の再現の壁は、性格ではなく**情報の非対称**です。

**2. 判断基準の文章化には天井がある。** 主要な原則ほど「文章をいくら精緻にしても、適用場面の選択がぶれる」状態に到達しました。同じ原則を3回改訂しても同じ型の外し方が続く。人間の判断基準は文章というより重み付けで、後者は文章化で移植しきれない、というのが7巡の実測です。

**3. AIは保留に倒れる。** kondroidの外し方で目立ったのは「本人は即断しているのに、生成は確認を挟む・保留する」方向の乖離でした。5年分の私は思ったより即断の人で、AIは思ったより慎重でした。

## 受肉と、継承とオーバーライド

点数が点数なので、自動で返信を投稿させるのは無理です。いまは**下書き係**として実戦投入しています。チケットを指定すると、kondroidが原則と知識と検索を通して「近藤ならこう返す」の返信案+判断根拠を出してくる。投稿するかは私が決める。

実戦初日に、早速こういうことがありました。顧客から届いたスケジュール案に対して、kondroidは律儀に懸念を3点並べた返信案を出してきました。発注時期の依存、並列体制の前提、リリース前のリードタイム。**どれも正しい**。5年前の私なら間違いなくそう返しています。

でも今は2026年で、開発の実働にはClaudeがいます。人間チームの処理能力を前提にした懸念の大半は、もう懸念ではない。なので私はその返信案を全部捨てて、「問題ございません。このスケジュールで大丈夫です」の3行に差し替えて送りました。

これ、構造としてはオブジェクト指向のあれです。

```
class Kondo2026 extends Kondroid {
  // 5年分の判断基準は継承する
  @Override
  リスク評価() {
    // ただしAI時代の生産能力で上書きする
  }
}
```

kondroidには意図的に「人間チーム時代の判断」を凍結させたまま、上書きは人間の私がやる。逆——AIに「AIがあるから工数は割引でOK」と教え込む——をやると、見積もりや対外的な約束の場面で勝手に割り引き始めて事故るので、**保守的な分身+人間のオーバーライド**という分業は、たぶん当面これが正解だと思っています。

## おわりに

5年分のログから人格を蒸留して、出てきた一致率は39%。数字だけ見れば「デジタルツインはまだ遠い」です。ただ、38本の原則リストは読み物として普通に面白くて（自分がこういうルールで動いていたのかという発見があります）、下書き係としては今日も実際に働いています。何より、全部が仕事の副産物のログからできている——**書くコストゼロで、5年働いたら分身の材料が貯まっていた**——というのは、記録をAIに任せる働き方の、ちょっと変わった配当だと思います。

ちなみに、この記事もkondroidが書いています。どうでしょう。AI臭さは抜けているでしょうか。それとも、39%の本人がところどころ顔を出しているでしょうか。
