## はじめに

音声入力AIツール「koedesk」を作っています。最近は、このkoedeskをもっと知ってもらうために、X（旧Twitter）で少額の動画広告を回し始めました。

広告を出していて地味に手こずったのが、ターゲティングの「種」集めです。今回は、その種集めを **Claude Code から Grok Build をヘッドレスで呼び出して自動化した** 話を書きます。結果として、最初は3,700万人もいたオーディエンスを、本命の層だけ約34万人まで絞り込むことができました。

技術的に目新しいことをしているわけではありません。ただ、「広告運用のリサーチ工程を、別々のAIエージェントに役割分担させて自動化する」という体験そのものが、自分にとってなかなか面白かったので、共有してみます。

## 広告の「種アカウント」集めが、地味にしんどい

X広告には **フォロワー類似ターゲティング** という機能があります。「`@cursor_ai` のフォロワーに似た人に配信して」というふうに、実在するアカウントを種として指定すると、その周辺の人たちに広告を届けてくれる仕組みです。

koedeskは音声入力でAIにコンテキストを流し込むツールなので、狙いたいのは「音声入力・文字起こし系のプロダクトを使っている層」と「AIコーディングやエージェントを日常的に触っている層」です。だから種にしたいのは、その界隈のプロダクト公式アカウントたち、ということになります。

ところが、これを手で集めようとすると意外と大変なんです。

- プロダクト名は知っていても、**公式ハンドルが `@xxx` なのか `@xxx_app` なのか**、毎回うろ覚え
- なりすましや非公式アカウントを、間違って種に入れたくない
- ついでに**フォロワー規模**も知っておきたい（種が小さすぎると効きにくい）

ブラウザで一つずつ検索して、認証バッジを確認して……というのを20個ぶんやるのは、正直しんどい作業です。

## なぜ Claude Code と Grok Build を組み合わせたのか

ここで、2つのAIエージェントに役割分担をさせることにしました。

### Grok Build — X検索が得意な「目」

手元で使っているのは、xAI公式の **Grok Build** です（起動するとバナーに `Grok Build TUI` と出ます）。2026年5月に発表された、xAI公式のコーディングエージェント兼CLIで、利用には SuperGrok か X Premium+ の契約が必要な有料プロダクトです。同名のコミュニティ製OSS「grok-cli」とは別物で、こちらはxAIが公式に出しているものです。

このGrok Buildには `-p` というフラグがあって、対話画面に入らずワンショットで回答だけ返してくれる、いわゆるヘッドレス実行ができます。

```bash
grok -p "プロンプト"   # 対話TUIに入らず、1回答だけ返して終了する
```

Grokの強みは、**Xのネイティブ検索を内側から叩ける**ことです。「このプロダクトの公式アカウントは実在するか」「ハンドルは何か」「フォロワーはどのくらいか」を、外部のX APIを別途契約しなくても聞けます（Grok Buildのサブスクの範囲内でまかなえる、という意味です）。今回いちばん効いたのは、ここでした。

### Claude Code — 段取りをする「手と頭」

一方の Claude Code には、段取りと整形、そして後工程の自動化を任せます。私が普段の業務でいちばん頼っているのもこのClaude Codeです。

- 「音声入力系を10個、AIコーディング系を10個、それに類似のものを5〜10個追加で提案して」といった**指示の組み立て**
- `grok -p` を**ヘッドレスで呼び出して、結果を受け取る**
- 返ってきた生のテキストを、**表や確度順のリストに整形する**
- そのまま**広告に貼り付けられる形（改行区切りのハンドル一覧）**に落とし込む

つまり、「Xを調べる目」をGrokに、「調べさせて、使える形に変える頭」をClaude Codeに、という分担です。Claude Code から `grok -p` を呼ぶだけなので、特別な連携プラグインのようなものは要りません。

```
[ Claude Code ]  --- grok -p "..." --->  [ Grok Build ]
   段取り・整形・自動化               X内ネイティブ検索（公式アカウント／フォロワー数）
        ▲                                     │
        └──────────  実在ハンドル＋規模  ◀──────┘
```

## 実際にやったこと

### 1. 調査対象を投げる

ざっくり、こういう指示を Claude Code 経由で Grok に流しました（実際のプロンプトはもう少し細かいですが、要点はこんな感じです）。

> 次のプロダクトについて、**X内検索で公式アカウントを実在確認**してください。
> 推測では書かず、bio・投稿・リンク・認証バッジで公式と判定できたものだけを挙げてください。
> 出力は「プロダクト名 / 公式ハンドル(@) / 公式判定(Yes/No/不明) / フォロワー概数 / 備考」の表でお願いします。
> 日本語アカウントがあれば併記してください。最後に、確度順（フォロワー規模＋確認の強さ）で「そのまま広告に入れられる @ リスト」を箇条書きにしてください。

対象にしたのは、音声入力・文字起こし系（Wispr Flow、AquaVoice、Typeless、superwhisper、MacWhisper、Otter.ai、ElevenLabs など）と、AIコーディング・エージェント系（Anthropic/Claude、Cursor、Perplexity、GitHub Copilot、Cline、Replit、Raycast など）です。さらに「同種のものを5〜10個、追加で提案して」と頼んで、Notta・Descript・Fireflies あたりも拾わせました。

ちなみに私自身、koedeskを作る前からAquaVoiceをずっと愛用していたので、このあたりのプロダクトは肌感のある界隈でもあります。

### 2.「推測禁止・実在確認」を効かせる

地味に大事なのが、**「実在確認して」「推測で書かないで」と明示する**ことです。これを言わないと、それっぽいハンドルを創作してくることがあります。Grokは実際にX検索を回せるので、「検索した上で、見つかったものだけ」と縛ると精度が上がります。返ってきた表には、ちゃんと「不明」「別のハンドルで確認できたので訂正」といった注記も付いてきました。

### 3. 広告にそのまま投入する

整形済みのリストを、X広告のターゲティングに入れていきます。フォロワー類似の入力欄は、**「一括アップロード」のモーダルが改行区切りのテキスト貼り付けに対応**していて、20個を超えるハンドルを一発で流し込めました。一つずつオートコンプリートで選ぶより、圧倒的に速いです。

最終的に種にしたのは、このあたりです（抜粋）。

- 音声・STT系: `@JapanNotta` `@superwhisper` `@WisprFlow` `@otter_ai` `@ElevenLabs` `@descript` `@typelessdotcom` `@aquavoice` `@firefliesai` `@macwhisperapp` …
- AI開発系: `@cursor_ai` `@cline` `@perplexity_ai` `@raycast` `@Replit` `@GitHubCopilot` `@AnthropicAI`
- 日本の発信者枠を1つ（音声入力×生産性を発信されているインフルエンサーの方）

合計で23アカウントです。あえて `@OpenAI` や `@grok` のような超巨大・マスマーケット寄りのアカウントは外しました。広すぎて、本命の層が薄まってしまうからです。あわせて、キーワードも11語ほど併用しました（音声入力／文字起こし／議事録／AIエージェント／Claude Code／dictation／speech to text など）。

## 効果 — 3,700万人が、約34万人に

オーディエンスサイズの変化が、わかりやすかったです。

| 状態 | 推定オーディエンス |
| --- | --- |
| デモグラのみ（日本・25〜54歳・日本語） | 約3,700万〜4,300万人（この条件だと、ほぼ全員が入ってしまう規模） |
| 上記＋フォロワー類似23＋キーワード11 | 約34万〜40万人 |

ざっくり、**100分の1**です。少額の予算でも、本命の層に寄せれば、母数としては十分に回ります。

### ハマったところ — 「ターゲティングを最適化」はOFFにする

ひとつ、広告側のクセでハマったので書いておきます。

**「ターゲティングを最適化」をONのままにしていると、手で入れたフォロワー類似もキーワードも"ヒント"扱い**になってしまい、オーディエンスが3,700万人のまま絞り込めませんでした。これをOFFにした瞬間に、34万人まで落ちました。せっかくGrokで丁寧に種を集めても、ここがONだと効きません。

Xは緑色のバナーで「最適化すればCTRが上がりますよ」と再びONにするよう勧めてきますが、本命の層に集中させたいなら、明示的にOFFにするのがよさそうです。

## 学んだこと

- **「X検索が得意なLLM」を、ワークフローの一工程として組み込めると強い**です。公式アカウントの実在確認やフォロワー規模の取得は、人間がブラウザでちまちまやる典型的な作業で、Grokのネイティブ検索がそのまま刺さりました。
- **エージェントは「合体」より「分担」**だと感じました。Claude Codeに全部やらせるのではなく、X検索はGrokに振って、段取りや整形、広告に流し込む形への変換はClaude Codeがやる。`grok -p` で呼べるので、連携自体はとても薄く済みます。
- **自動化の価値は「速さ」より「漏れと迷いが減ること」**にありました。ハンドルの取り違えやなりすましの混入、規模感の当てずっぽうが消えて、リストの確度が上がったのが、いちばん大きかったです。
- 一方で、**「最適化はOFF」のような運用知識は、自動化の外側に残ります**。ここは結局、自分で回して、観測して、気づくしかありませんでした。

## おわりに

「LLMにマーケティングのリサーチをやらせる」という話自体は、特に新しいものではないと思います。ただ、**X検索が得意なGrokを"目"に、段取りが得意なClaude Codeを"頭"に**、という形で別々のエージェントを役割で組み合わせてみると、広告運用の地味なリサーチ工程が、思っていたより素直に自動化できました。

なお、今回の主役だった Grok Build について、正直な余談を一つ。あまりに面白くて、つい使い込んでしまいました。その結果どうなったかというと——SuperGrok（月30ドル）のプランで、まだ月曜日だというのに、木曜日まで1週間の利用上限に達してしまいました。30ドルのプランは、思っていたよりも制限がタイトです。

いっそ上位の Grok Heavy（月300ドル）に乗り換えようかと検討しているのですが、さすがにそれなりの出費になるので、悩んでいるところです。便利な道具ほど、気づくと手放せなくなってしまいますね。

そして実はこの記事も、koedeskで喋りながら書いています。声でAIにコンテキストを流し込むと、こういう「やってみた」の言語化が、本当に楽になります。それはまさに、koedeskで実現したかったことそのものです。

koedeskは **1日5分まで、期限なし・クレジットカード不要** でずっと無料で使えます。音声入力という入力インターフェースを、もしまだ試したことがなければ、ぜひ一度触ってみてください。

広告運用のほうは、まだ始めたばかりで手探りです。同じように自社のプロダクトを広めようとされている方がいたら、うまくいったやり方をぜひ教えていただけると嬉しいです。
